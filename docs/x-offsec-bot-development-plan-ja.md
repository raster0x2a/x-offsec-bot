# X向けオフェンシブセキュリティ情報収集AI Bot 開発プラン（GitHub Actions + GitHub Pages無料運用版）

## 1. 目的と成功条件

### 目的
- X（旧Twitter）から**毎日1回**、オフェンシブセキュリティに関する情報（ツール紹介、技術ブログ、ニュース）を自動収集する。
- 収集したリンクを重複除去・分類・短文要約し、**GitHub Pages上のWebサイト**で「プレビュー + 簡単なサマリ」付き一覧として公開する。
- 可能な限り無料枠で運用する。

### 成功条件（MVP）
- GitHub Actionsのcronで日次実行される（例: UTC 00:00 = JST 09:00）。
- 収集件数（例: 100〜300件/日）を安定達成する。
- 重複除去後の有用候補10〜30件を日次ページに表示する。
- 各アイテムにカテゴリ、要約、リンク先メタ情報（タイトル/説明）を表示する。
- 失敗時はIssue作成またはメール通知で検知できる。

---

## 2. 無料運用を前提とした構成

### 採用方針
- **実行基盤**: GitHub Actions（無料枠）
- **ホスティング**: GitHub Pages（静的サイト）
- **データ保存**: Git管理（JSON/Markdown）
- **通知**: GitHub Issue作成（無料）
- **LLM要約**: 初期はルールベースを主、必要時のみLLM（コスト制御）

### コスト最適化ルール
1. まずルールベースでスコア上位のみ抽出（例: 20件）
2. LLM要約は上位のみ実行（例: 10件）
3. 既出URLは再要約しない（キャッシュ）
4. 失敗時リトライは1回まで（無限再試行しない）

---

## 3. スコープ定義

### 収集対象
- セキュリティ研究者、Red Team、Bug Bounty、CTF、Exploit/PoC関連アカウント投稿。
- 共有リンク先（ブログ、GitHub、ニュース、アドバイザリ、CVE）。

### 除外対象
- 違法行為を助長する投稿。
- スパム・広告のみの投稿。
- 同一URLの再掲や、同一話題の短時間大量投稿。

### 初期カテゴリ
- **Tool**: 新規/更新ツール、GitHubリリース、PoC
- **Blog/Writeup**: 技術解説、検証記事、脆弱性分析
- **News/Advisory**: CVE、ベンダーアドバイザリ、インシデント関連

## 4. 想定アーキテクチャ（静的サイト公開）

1. **Collector（Python）**
   - X APIで検索/リスト取得。
   - 投稿テキスト、投稿URL、外部リンク、時刻、著者を抽出。
2. **Normalizer**
   - URL展開、トラッキングパラメータ除去、ドメイン正規化。
3. **Deduplicator & Ranker**
   - URL一致 + タイトル類似度で重複統合。
   - 鮮度・信頼ソース・反応量でスコアリング。
4. **Summarizer**
   - ルールベース要約を基本に、必要に応じてLLM補完。
5. **Page Builder**
   - `data/daily/YYYY-MM-DD.json`を生成。
   - テンプレートから`docs/`配下の静的HTMLを生成。
6. **Feedback Tuner（バックエンドなし）**
   - GitHub Issue/PRコメント/リアクションを評価データとして集計。
   - `data/config/feedback_rules.json` を更新し、翌日の収集重みへ反映。
7. **Publisher（GitHub Pages）**
   - Actionsが生成物をコミットし、Pagesに自動反映。

---

## 5. データフロー（日次バッチ）

1. 直近24時間の対象ポスト収集
2. 外部リンク抽出と正規化
3. OGP/メタ情報取得（タイトル/description/og:image）
4. 重複除去とカテゴリ分類
5. 上位N件の短文要約（2〜3行）
6. 前日までの評価データ（Issue/リアクション）を読み込み、重み調整
7. `daily.json`と`index.html`を生成
8. 生成物をコミットしGitHub Pagesへ公開
9. 失敗時はIssue作成

## 8.1 ポスト選定の評価と傾向フィードバック（バックエンドなし）

### 結論
- **実現可能**。GitHub上の機能だけで、収集結果の評価→翌日の重み調整まで回せる。

### 方式A（推奨）: Issueベース評価
1. 日次ページの各アイテムに「良い/不要」リンクを表示（IssueテンプレートURL）。
2. ユーザーはリンク先で評価を送信（`good` / `noise` ラベル）。
3. 毎日実行のActionsが前日Issueを取得して集計。
4. `feedback_rules.json` の重みを更新（例: 高評価ドメイン加点、低評価キーワード減点）。
5. 翌日の収集・ランキングで新しい重みを利用。

### 方式B: PRレビュー評価
1. 日次データJSONをPRとして自動作成。
2. レビューコメントやリアクションを収集してスコア化。
3. スコアを重みに反映して翌日へ適用。

### 反映できるパラメータ例
- ドメイン重み（例: `trusted_domains` の加点）
- 著者重み（例: 高評価著者の優先）
- キーワード重み（例: ノイズ語の減点）
- カテゴリ比率（Tool/Blog/Newsの採用バランス）

### バックエンドなし運用のポイント
- 状態はすべてリポジトリ内のJSONで管理（Git履歴で監査可能）。
- 集計はActionsの定期ジョブのみで実行。
- UIは静的HTML + GitHubリンクのみ（サーバレス）。
- 不正入力対策として、評価をorgメンバーのIssueに限定可能。

### 制約
- リアルタイム反映は不可（次回バッチ時に反映）。
- 評価者が少ない場合は重みが偏るため、最小サンプル数しきい値を設ける。

## 13. リスクと対策

1. **X API制限・仕様変更**
   - 検索とアカウント監視を併用し、取得元を冗長化。
2. **無料枠超過**
   - 取得件数上限・要約件数上限・再処理禁止で制御。
3. **ノイズ投稿混入**
   - 除外キーワード、低信頼ドメイン除外、著者スコア導入。
4. **Pages反映失敗**
   - 失敗時にIssue自動作成、手動再実行手順をRunbook化。
5. **評価バイアス**
   - 最小投票数しきい値、週次で重み変動上限を設定。
